# 🚩 初期タスクガイド (First Tasks)

このドキュメントでは、開発環境の構築後に最初に取り組むべきタスク（Good First Issue）の進め方と、実際のコード変更からリリースにいたるまでのステップを解説します。

---

## 🔍 1. 初期タスク (Good First Issue) の見つけ方

チームでは、新メンバーがシステムのアーキテクチャやデプロイフローに慣れるため、影響範囲が小さく、作業手順が分かりやすい課題を用意しています。

### 課題の探し方
- **GitHub Issues / Jira ボード**:
  - `good-first-issue` または `onboarding` ラベルが付いたチケットを探してください。
  - チケットの一覧リンク: `[要記入: Jira/GitHub フィルタ済みのURL]`
- **タスクの例**:
  - 管理画面の文言変更やスタイルの微修正
  - 特定のコンポーネントに対するユニットテストの追加
  - 不要になった古いコードやアセットの削除・クリーンアップ

> [!TIP]
> どの課題に着手するか迷った場合は、メンターに相談して最適なアサインを決めてもらいましょう。

---

## 🛠️ 2. タスク着手からリリースまでの流れ (実戦演習)

開発環境セットアップを終えたら、実際に以下の手順で最初の変更を本番環境へデプロイしてみましょう。

### Step 1: 課題の担当者に自分を設定する
- Jira または GitHub で対象のチケットステータスを `[進行中 / In Progress]` に変更し、アサイニーに自分を設定します。

### Step 2: 開発ブランチの作成と実装
```bash
# 最新のmainブランチを取り込む
git checkout main
git pull origin main

# 新しい機能開発用ブランチを作成
git checkout -b feature/TICKET-123_fix-button-style
```
- コードを変更し、ローカルで動作を確認します。
- 必要に応じてユニットテストを実行・追加します。
  ```bash
  npm run test # テスト実行コマンド
  ```

### Step 3: プルリクエスト (PR) の作成とレビュー
- 変更をリモートリポジトリにプッシュします。
- GitHub上でPRを作成し、以下を記述します：
  - **解決する課題**: チケットのリンク
  - **変更内容**: どのような変更を行ったか
  - **テスト内容**: どのように動作確認したか
- メンターやチームのメンバーにレビューを依頼します。

### Step 4: ステージング環境での確認
- レビューが承認され、コードが `develop` または `main` ブランチにマージされると、ステージング環境に自動デプロイされます。
- ステージング環境 `[要記入: ステージングURL]` にアクセスし、自分の変更が正しく動いているか確認（QA）します。

### Step 5: リリース
- 週次のリリースサイクルに従って、自分の変更が本番環境へデプロイされるのを確認します。
- 本番デプロイ完了後、実際に本番環境でも正しく動作しているか確認し、チケットを `[完了 / Done]` に変更します。

---

## 💡 3. スムーズに進めるためのアドバイス

1. **まずは仕様よりも「流れ」を体験する**:
   - 最初のタスクの主目的は「コードを書いて、レビューを受けて、本番に届く」までの一連の開発プロセスを体験することです。コード規模は数行の変更で十分です。
2. **自己レビューのススメ**:
   - PRを出す前に、GitHubの差分（Diff）を自分で一行ずつ見直してみてください。不要なコンソール出力や、意図しないインデントのズレを防ぐことができます。
3. **ドキュメントの修正も大歓迎**:
   - このオンボーディングプロセス中に「手順が古くなっている」「記載が足りない」と感じた点があれば、それ自体を修正するPRを出すのも素晴らしい初期タスクになります。
